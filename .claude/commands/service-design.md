# Service Design — 설계 원칙 수립

> **guided 모드**에서는 AI가 코드를 먼저 읽고 **선택지를 제시**한다. 사용자는 고르기만 하면 된다. 비개발자도 사용 가능.
>
> **full 모드**는 자유 기술 인터뷰. 개발자/기획자용.
>
> 두 모드 모두 코드를 수정하지 않는다. 출력은 `docs/design/service-principles.md` 한 파일.
>
> **참조 문서**: 결정 지점에서 🟡🔴 지식 경고가 발생하면 `.claude/refs/service-design-knowledge-gaps.md`를 `Read` 도구로 참조하여 해당 개념의 해설을 사용자에게 제공한다. 이 파일은 커맨드 설치 시 함께 배치된다.

## 핵심 철학

> **"생성보다 인식이 쉽다. 코드가 말하는 것을 사용자가 확인하게 하라."**

이 커맨드는 세 가지 원칙을 지킨다:

1. AI는 서비스 도메인/로드맵을 모르므로 **사용자 확인이 반드시 필요하다**
2. 그러나 비개발자는 "엔티티를 나열하라" 같은 생성형 질문에 답할 수 없다 → **선택지 제시로 인지형 질문으로 전환**
3. 개발 지식이 필요한 결정은 **선택지 바로 옆에 🔵🟡🔴 태그**를 붙여 선택하는 순간 인지하게 한다

---

## 호출 조건

다음 중 하나에 해당하면 시작한다:

- 사용자가 명시적으로 `service-design` 요청
- 신규 프로젝트 초기에 방향을 잡고 싶을 때
- `architecture-review`가 원칙 문서 부재를 보고하며 선행 권장
- 기존 프로젝트의 로드맵이 크게 바뀔 때

---

## 모드 선택 (Phase -1)

| 모드 | 대상 | 방식 | 출력 |
|------|------|------|------|
| **guided** (기본) | 누구나 | AI 코드 스캔 → 선택지 제시 → 사용자 선택 | 전체 설계 원칙 문서 |
| **full** | 개발자/기획자 | 자유 기술 인터뷰 | 전체 설계 원칙 문서 (더 깊음) |

사용자가 명시하지 않으면 **guided가 기본**. full 모드는 사용자가 `service-design full 모드로` 명시해야만 시작.

## Phase 0 — 근거 수집 (guided 모드에서 필수)

guided 모드는 선택지가 **근거에 기반**해야 한다. Phase 0에서 AI가 다음을 스캔하여 증거를 모은다:

- 파일/폴더 구조 (도메인 힌트)
- 주요 타입/클래스/인터페이스 선언
- 외부 의존성 목록 (package.json / requirements 등)
- 라우팅/엔트리 포인트
- README, CLAUDE.md, 주요 주석
- (있다면) 기존 `service-principles.md`의 이전 버전

증거는 **"파일:라인 또는 파일명"** 형태로 기록해서, 각 Phase의 선택지에 근거를 바로 첨부할 수 있게 한다.

Phase 0 결과는 문서로 저장하지 않는다. 다음 Phase 선택지 생성의 입력으로만 사용.

### Phase 0 증거 부족 시 폴백

다음 조건에 해당하면 guided 모드의 선택지 생성이 불가능하다:

- 코드가 < 50줄 또는 폴더 구조가 템플릿 스캐폴드만
- 도메인 힌트 없이 `main.ts` / `index.html` 하나 수준
- README·주석도 비어 있음

이때 AI는 다음을 사용자에게 제시한다:

```
guided 모드의 선택지를 만들 만한 코드 증거가 부족합니다. 다음 중 선택:

[ ] full 모드로 전환 — 자유 기술 인터뷰 (개발자 경험이 있다면 권장)
[ ] 시작 단계 전용 미니 인터뷰 — 3 문항만 받고 간소 문서 생성
    (서비스 정체성 / 주 사용자 / 다음 기능 1개)
[ ] 지금 중단 — 코드가 조금 더 생긴 뒤 다시 돌리기
```

미니 인터뷰를 고르면 Phase 2-5를 건너뛰고 Phase 1 3 문항만 받아 1페이지짜리 `service-principles.md`를 생성한다. 결정 섹션은 "현재 구조 유지" 하나만.

## 선택지 제시 프로토콜 (guided 모드 — Phase 1~5 공통)

각 Phase마다 AI가 다음을 수행:

**1. 증거에서 2-4개 선택지 생성**

- 너무 적으면(1개) 선택이 아니라 예/아니오 — 선택지 2개 이상 필수
- 너무 많으면(5+) 결정 피로 — 최대 4개
- 각 선택지는 **plain language** 한 줄 + **증거** 한 줄

**2. 각 선택지에 지식 태그 부착 (해당 시)**

- 🔵 배경 개념 있음 (참고용)
- 🟡 선택 시 이 개념을 어느 정도 이해해야 함
- 🔴 선택 시 이 개념을 모르면 설계가 깨질 수 있음 — KNOWLEDGE-GAPS.md 참조 권장

**3. 마지막 선택지는 항상 탈출구**

- `[ ] 이 중 없음 — 직접 설명할게요` (자유 입력으로 폴백)
- `[ ] 잘 모르겠음 — AI가 추정하고 ⚠️ 임시로 표시` (임시 결정으로 진행)

**4. 사용자 선택 수령 후 마킹**

- 선택됨 → `[선택됨]`
- 자유 입력 → `[직접 입력]`
- AI 추정 → `[AI 추정] ⚠️`
- service-principles.md에 출처 태그와 함께 기록

**5. AI 추정 남용 방지**

사용자가 **3개 Phase 연속**으로 "잘 모르겠음 — AI 추정"만 선택하면 AI가 **일시 정지**하고 경고:

```
⚠️ 지난 3개 Phase에서 제가 단독으로 추정해왔습니다.
이대로 가면 결과물이 실제 서비스와 맞지 않을 가능성이 큽니다.

[ ] 이대로 계속 — 결과물을 "참고용 초안"으로만 사용하겠음
[ ] 처음부터 다시 — 제가 놓친 정보를 알려주시겠어요?
[ ] 인터뷰 중단 — 코드/기획이 더 구체화된 뒤 다시 돌리기
```

이 안전장치가 있어야 "최소 저항 경로"로 흘러가서 무의미한 문서가 생산되는 걸 막을 수 있다.

## Phase 1 — 서비스 정체성

### guided 모드

```
[코드/README에서 추론한 초안]
제가 이 서비스를 이렇게 이해했습니다:

[ ] 초안 A: "{AI가 제안한 한 줄 정의 A}"
    근거: README.md / 주요 라우트 / 핵심 타입
[ ] 초안 B: "{AI가 제안한 한 줄 정의 B}"
    근거: ...
[ ] 이 중 없음 — 직접 설명할게요
```

주 사용자, 주요 동기도 같은 방식으로 2-3개 선택지 제시.

### full 모드

자유 기술. 답이 5문장 이내로 나오는지 확인하고, '누구를 위해'와 '무엇을 하는가'가 모두 명확한지 판정한다. 불명확하면 3회까지 되묻되, 그 뒤에도 흐릿하면 **인터뷰 중단**하고 "먼저 서비스 정체성부터 정리하자" 권고.

질문:

1. "이 서비스를 한 줄로 정의하면? `____를 위해 ____를 하는 도구` 형태로."
2. "핵심 사용자 페르소나 1-2개. 각 페르소나의 맥락(역할, 숙련도, 사용 환경)까지 간단히."
3. "사용자가 이 서비스에 오는 주된 이유 1-3개. 각각의 빈도/중요도도."
4. (선택) "유사 서비스가 있다면? 이 서비스가 그것과 다른 점은?"

## Phase 2 — 도메인 모델

### guided 모드

```
코드에서 다음이 엔티티로 보입니다.
'고유 정체성을 가지는 것'(예: 저장되고 ID로 추적되는 것)을 골라주세요. 여러 개 OK.

[ ] User — 사용자 계정 (auth.ts, user-store.ts에서 발견)
[ ] Todo — 할 일 항목 (todo.ts, list-view.ts)
[ ] Category — 분류 (category-picker.tsx)
[ ] Tag — 태그 (tag-filter.ts)
[ ] 이 중 없음 — 직접 설명할게요
```

관계와 생애 주기는 엔티티 확정 후 별도 선택지:

```
User와 Todo의 관계는?
[ ] User 1 — N Todo (한 사용자가 여러 할 일)
[ ] Todo가 User를 모름 (익명/공유)
[ ] 복잡 — 직접 설명할게요
```

### full 모드

자유 기술. 아래 항목이 모두 채워져야 Phase 3으로 진행.

1. "서비스가 다루는 '실제 세계의 것들' 3-7개를 나열하세요. 각각이 **엔티티**인지(ID로 추적됨, 생애 있음) **값 객체**인지(값 자체로 정의) 구분해서."
2. "엔티티들 사이의 관계(1:1 / 1:N / N:M)와 방향(소유 / 참조)."
3. "각 엔티티의 생애 주기. 언제 만들어지고, 어떤 상태 전이가 있고, 언제 사라지는가. 상태가 5개 이상이면 상태 다이어그램 스케치."
4. "도메인 경계(bounded context)가 2개 이상 있는가? 있다면 각 경계의 이름과 내부 엔티티."

사용자가 엔티티/값 객체 구분을 못하면 🟡 경고하고 `KNOWLEDGE-GAPS.md#domain-model` 안내.

## Phase 3 — 현재 기능 인벤토리

### guided 모드

```
코드에서 다음 기능이 보입니다. 현재 제공 중인 것을 골라주세요 (여러 개):

[ ] 할 일 추가 (todo-form.ts)
[ ] 할 일 체크/완료 (todo-item.ts)
[ ] 카테고리 필터 (filter-bar.tsx)
[ ] 검색 (search-input.ts)
[ ] 로컬 저장 (localStorage-store.ts)
[ ] 이 중 없는 것이 더 있다 — 알려주세요
```

기능 수가 < 5이면 AI가 "설계를 걱정할 단계가 아닐 수 있다. 지금 멈추고 기능을 더 만든 뒤 돌아와도 된다"를 알려준다.

### full 모드

자유 기술.

1. "지금 제공 중인 주요 기능 5-10개를 나열. 각 기능을 동사+목적어 형태로(예: '주문 생성' / '재고 확인')."
2. "각 기능이 조작하는 엔티티 및 발생시키는 주요 이벤트."
3. "여러 기능이 공통으로 참조하는 자원(공통 상태/공유 유틸/반복되는 검증 규칙) 목록. 3개 이상이면 이후 패턴 선택에 영향."
4. "기능 간 호출 관계 — A 기능이 B 기능을 직접 호출하는 경우가 있는가? 있다면 그 방향."

기능 수 < 5이면 "아직 설계를 걱정할 단계가 아닐 수 있다. 기능을 더 만든 뒤 돌아와도 된다"를 안내하고 **인터뷰 중단 옵션** 제공.

## Phase 4 — 로드맵

### guided 모드

```
앞으로 추가할 수도 있는 기능들을 몇 가지 제안드립니다. 해당하는 것에 체크:

[ ] 실제로 추가하려는 것
[ ] 🤔 고민 중인 것  
[ ] ❌ 할 생각 없음

후보:
[ ] 사용자 로그인 (현재 익명 → 계정 기반)
[ ] 데이터 동기화 (로컬 → 서버)
[ ] 🔴 실시간 협업 (선택 시 일관성 모델 학습 필요)
[ ] 모바일 지원
[ ] 그 외: (직접 입력)

이 중 아무것도 당장 없으면 "없음" 체크 OK — "현재 구조 유지" 원칙만 생성됩니다.
```

🔴가 붙은 항목을 체크하면 AI가 즉시 안내:

```
🔴 실시간 협업을 선택하셨습니다. 이 결정은 일관성 모델(KNOWLEDGE-GAPS.md#consistency-model)을 어느 정도 이해해야 설계가 흔들리지 않습니다.
- [선택 1] 지금 5분 읽고 올게요 → 인터뷰 일시 중단
- [선택 2] 일단 기본값(최종 일관성)으로 진행하고 ⚠️ 임시 결정으로 표시
- [선택 3] 이 기능은 빼겠습니다 → 선택 취소
```

### full 모드

자유 기술.

1. "앞으로 3-6개월 안에 추가하려는 기능 3-7개를 시기순으로."
2. "각 기능이 지금의 엔티티/이벤트에 미치는 영향 — 새 엔티티 추가? 기존 엔티티 변형? 새 관계? 새 이벤트?"
3. "불확실한 기능(넣을지 말지 모름)은 별도로 표시하고 확실도(%) 명시."
4. "로드맵에서 가장 구조를 뒤흔들 가능성이 큰 기능 Top 1 선정. 이 기능이 지금의 코드에 얼마나 겹치는지 감각적으로."

사용자가 "로드맵 없다"고 답하면 🟡 경고:

```
🟡 로드맵 없이 진행하면 설계가 '지금 상태'에만 맞춰져, 나중에 기능 추가 시 원칙을 다시 돌려야 합니다.
- [선택 1] "그래도 추가하고 싶은 것 하나"만 적기 (권장)
- [선택 2] 로드맵 없이 진행 — "현재 구조 유지 원칙"만 생성
```

## Phase 5 — 확장 축과 제약

### guided 모드

```
앞으로 복잡도가 커질 방향을 1-2개 골라주세요:

[ ] 기능 종류가 늘어날 것이다 (가장 흔함)
[ ] 사용자 수가 늘어날 것이다
[ ] 🔴 동시 협업/실시간이 생길 것이다 — 일관성 모델 학습 필요
[ ] 🔴 모바일/다기기 지원이 생길 것이다 — 동기화 전략 필요  
[ ] 지금 잘 돌아가게만 하고 싶다 — 확장 생각 안 함
```

제약 사항도 같은 방식으로 순차 체크박스 — 스택(정해짐 락인 / 정해짐 가변 / 미정), 팀 규모(혼자 / 2-3명 / 4명+), 배포 환경(정적 호스팅 / 서버리스 / 자체 서버 / 미정). AI는 `package.json`, deploy 스크립트, config 파일 등에서 추론한 값을 각 선택지 옆에 근거로 붙인다.

### full 모드

자유 기술.

1. "앞으로 복잡도가 커질 방향(확장 축) 1-2개. 후보: 사용자 수 / 컨텐츠 종류 / 동시 협업 / 실시간성 / 디바이스 종류 / 기타. 2개 초과 선택 시 초점이 흐려진다."
2. "기술 스택 자유도. 정해진 스택이 있는가? 락인 수준은?"
3. "팀 규모와 숙련도. 아키텍처 결정에 **팀이 소화 가능한 복잡도**를 맞춰야 한다."
4. "배포 환경 제약. 정적 호스팅 / 서버리스 / 자체 호스팅 / 모바일 앱 / 엔터프라이즈 온프레미스 등."
5. "예산/시간 제약. 몇 개월에 걸쳐 구축 가능한가."
6. (선택) "비기능 요구사항 — 성능 목표, 가용성 목표, 접근성, 보안 등급."

🔴 경고: 확장 축으로 "실시간성" 또는 "동시 협업" 선택 시 일관성 모델 학습 필요. `KNOWLEDGE-GAPS.md#consistency-model` 안내하고 임시 결정 ⚠️로 진행할지 확인.

---

## Phase 6 — 지식 경고 종합 (간소화됨)

선택지 제시 프로토콜로 인해 지식 경고가 **선택 시점에 이미 표시**되었다. 이 Phase의 역할은 단순화됨:

1. Phase 1-5에서 마킹된 🔴 선택들을 수집
2. 그중 "임시 결정 ⚠️"으로 진행된 항목을 후속 검토 리스트로 집약
3. ⚠️ 항목이 5건 이상이면 사용자에게 최종 확인:

```
⚠️ 임시 결정이 {N}건 누적되었습니다. 다음 중 선택:
- [선택 1] 이대로 문서 생성, 후속 검토 리스트 포함 (권장)
- [선택 2] 가장 중요한 2-3건만 지금 학습하고 돌아와서 확정
- [선택 3] 전부 학습하고 돌아오겠음 (며칠-몇 주)
```

## 결정 지점과 지식 경고 매핑 (reference)

| 결정 | 필요 지식 | 기본 레벨 |
|------|-----------|-----------|

## architecture-review와 intent-review와의 관계

- **service-design** (이 스킬): 원칙을 정한다. guided 모드는 코드를 읽고 사용자는 확인만.
- **architecture-review**: 원칙에 맞게 코드 경로를 진단한다.
- **intent-review**: 원칙/구조에 따라 생긴 함수들의 이름을 검증한다.

순환 관계:

```
service-design → principles.md
     ↓
architecture-review (principles.md 참조, 없으면 Phase 0.5 자동 추론)
     ↓
intent-review
     ↓
(큰 로드맵 변화 시) service-design 재실행
```

`principles.md`가 있으면 두 후속 스킬이 **자동 참조**. 없으면 architecture-review가 Phase 0.5 자동 추론 컨텍스트로 대체한다 (1회성).

### guided vs Phase 0.5의 차이

- **Phase 0.5 (architecture-review)**: 이번 진단 한 번에만 쓰는 휘발성 컨텍스트. 파일 안 남음.
- **guided (service-design)**: `docs/design/service-principles.md` 영속 생성. 이후 모든 스킬의 판단 기준.

"이 프로젝트 하나 점검만" → Phase 0.5.
"이 프로젝트 오래 갈 거야" → service-design guided로 원칙 문서 고정.
