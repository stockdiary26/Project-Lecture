# Architecture Review — 데이터 흐름/이벤트 경로 진단

> intent-review의 상위 층. 함수 단위가 아닌 **관계** 단위의 문제를 다룬다.
>
> **이 커맨드는 진단과 방향 제안까지만 한다.** 실행 가능한 단계별 plan, 커밋 계획, Golden Master 시나리오 구체화는 별도 플랜 작성 스킬(예: `superpowers:writing-plans`)에 위임한다. 절대 코드를 수정하지 않는다.
>
> **service-principles.md 참조**: `docs/design/service-principles.md`가 존재하면 우선 참조. 없으면 Phase 0.5 자동 추론 컨텍스트로 진행.

## 핵심 철학

> **"상태는 흘러야 한다. 흐름이 없는 상태는 중복되고, 중복된 상태는 어긋난다."**

함수의 이름이 약속이듯, 경로의 이름도 약속이다. 이름 없는 경로로 흐르는 데이터는 동기화 버그와 의존 폭증의 근원이다.

---

## 호출 조건 (Precondition)

이 스킬은 다음 중 하나 이상이 성립할 때만 시작한다. 조건 미충족 시 intent-review 선행을 권장한다.

- 사용자가 명시적으로 `architecture-review` 요청
- intent-review 보고서의 "분산된 의도" 항목 ≥ 3건
- 같은 종류의 버그가 서로 다른 모듈에서 3회 이상 재발 기록
- 새 기능 하나 추가에 ≥ 4개 파일 수정이 필요한 상황

## 시작 전 확인 — 컨텍스트 소스 결정

시작 시 `docs/design/service-principles.md`의 존재를 먼저 확인한다.

- **있으면** (원칙 문서 기반 진단): 그 문서를 읽어 **판단 기준**으로 삼는다. 문서의 "결정 목록"과 "확장 축"이 패턴 선택의 우선 참조 기준이 된다. 임시 결정(⚠️ 태그)이 있으면 이주 제안 시 해당 영역은 보수적 기본값 유지.

- **없으면** (자동 추론 기반 진단 — 기본 경로): 바로 **Phase 0.5 — 컨텍스트 자동 추론**으로 진행한다. 이것이 **기본 경로**다. 소규모 프로젝트나 비개발자 사용자에게는 이 경로가 더 적합하다.

사용자에게 다음 안내를 간단히 표시:
> 원칙 문서가 없어 **자동 추론 컨텍스트**로 진행합니다. (3-5 문항 확인만 필요)
>
> 대규모/복잡한 프로젝트거나 팀 작업인 경우 `service-design`으로 원칙 문서를 먼저 만드는 것을 고려할 수 있습니다. 작은 프로젝트라면 지금 경로가 더 가볍고 충분합니다.

## Phase 0.5 — 컨텍스트 자동 추론 (원칙 문서 없을 때만)

이 단계는 **비개발자 사용자가 도메인 모델링 인터뷰를 하지 않고도 스킬을 쓸 수 있게 하기 위함**이다. AI가 코드를 읽고 초안을 만들고, 사용자는 최소한의 확인만 한다.

### 절차

**1. 코드 스캔 기반 초안 자동 생성**

AI가 다음을 읽어서 초안을 만든다:
- 파일/폴더 구조에서 주요 도메인 추론 (폴더명, 주요 클래스/타입, entry point)
- 외부 의존성 (package.json / requirements / Cargo.toml 등)
- 주요 UI 진입점 (라우트, 메인 컴포넌트)
- README/주석에서 서비스 목적 힌트

**2. 3-5 문항 확인 인터뷰**

질문은 간결하게, 선택지 위주로:

- "이 서비스를 한 줄로 뭐라고 설명하시겠어요? (초안: 제가 추론한 내용 → `{초안 문장}`. 맞으면 '맞음', 다르면 한 줄로 고쳐주세요)"
- "주 사용자는? (1-2개로 간단히)"
- "다음에 추가하고 싶은 기능이 하나라도 있나요? (없으면 '없음' OK)"
- "지금 가장 자주 건드리는 파일이나 영역은? (선택 사항)"

질문은 **3개 필수 + 2개 선택**. 사용자가 "모르겠다"로 답해도 진행 가능.

**3. 인라인 경량 컨텍스트 블록 생성**

- **별도 파일로 저장하지 않는다.** 이번 진단 한 번에만 사용.
- 이번 진단 보고서 상단에 포함.
- 추론한 항목에는 `[추론]`, 사용자 확인한 항목에는 `[확인됨]` 태그.

**4. Phase 1 진단으로 진행**

### 경량 컨텍스트 블록 템플릿

보고서 상단에 다음 블록을 삽입:

```markdown
## 컨텍스트 (자동 추론 + 사용자 확인)

- **서비스**: {한 줄 정의} [확인됨]
- **코드베이스 규모**: {파일 수, 주 언어} [추론]
- **주요 도메인(추정)**: {이름 목록} [추론]
- **주 사용자**: {답변} [확인됨]
- **다음 기능**: {답변 또는 "없음"} [확인됨]

> ℹ️ 이 컨텍스트는 이번 진단 한 번에만 사용됩니다. 누적된 원칙 문서가 필요해지면 `service-design` 스킬을 실행하세요.
```

### service-design 승격 권고 트리거

자동 추론 경로로 진행하되, 다음 조건 중 하나라도 해당하면 **보고서 끝에 service-design 전환 권고**를 포함한다:

- 사용자가 확장 축으로 "실시간성" 또는 "동시 협업" 방향을 언급
- 같은 코드베이스에 대해 architecture-review를 3회 이상 돌린 기록이 있음
- 임시 결정이 5건 이상 누적
- 결정 중 🔴 경고(일관성 모델 / 신뢰 경계 등)가 2건 이상 발생
- 사용자가 "팀"이라고 답함 (협업 환경)

권고 템플릿:

```markdown
## 다음 단계 권장

프로젝트가 자동 추론 컨텍스트만으로는 일관된 결정을 내리기 어려운 지점에 도달했습니다. 다음 라운드 전에 `service-design` 스킬로 원칙 문서를 만들어두면 이후 architecture-review와 intent-review가 일관되게 이 문서를 참조합니다.
```

### 자동 추론이 실패할 때

다음 경우에는 자동 추론을 포기하고 사용자에게 명시적 선택을 요청한다:

- 코드 스캔 결과 도메인 힌트가 전혀 없음 (빈 프로젝트/스캐폴드만)
- 사용자가 3 질문 모두 "모르겠다"로 답
- 파일/폴더 명이 너무 범용적이어서 추론이 무의미 (`src/`, `main.ts`만 있음 등)

이때의 안내:
> 코드만으로는 컨텍스트를 추론하기 어렵습니다. 다음 중 하나를 선택해 주세요:
> - [일반 원칙으로 진단] 컨텍스트 없이 보편 원칙만 적용 (결과가 덜 정확할 수 있음)
> - [service-design으로 전환] 짧은 인터뷰로 원칙 문서 생성 후 재진행

## 3단계 진단

### Phase 1: 상태 지도 (State Map)

대상 코드에서 **쓰기(write)** 지점을 먼저 찾는다.

- 할당, `push`, 필드 수정, DOM 직접 조작, `setState` 류, store 쓰기
- 각 쓰기의 "값의 정체성"(무엇을 나타내는가)을 기록

→ 같은 정체성의 값이 **≥ 2곳**에서 쓰여지면 **상태 중복** 의심.

### Phase 2: 이벤트 지도 (Event Map)

사용자 입력(키보드/클릭/스크롤/터치)과 시스템 이벤트(타이머/네트워크 응답/라이프사이클)의 진입점을 모두 나열한다.

→ **같은 이벤트가 여러 핸들러에서 각자 해석**되면 **명령 분산** 의심.

### Phase 3: 의존 방향 지도

컴포넌트/모듈 간 호출을 그래프로 정리한다.

- `A → B.internal` 형태의 직접 내부 접근이 **≥ 3건** → **결합 과다** 의심
- 순환 의존 존재 → 즉시 경고, 이번 라운드의 최우선 대상

---

## 증상 → 패턴 매핑

한 라운드에는 **하나의 패턴만** 제안한다. 증상이 겹치면 영향 범위가 가장 큰 하나를 선택하고, 나머지는 "대기 리스트"로 기록한다.

| 증상 | 제안 패턴 | 판정 힌트 |
|------|-----------|-----------|
| 동일 값이 ≥ 2곳에 저장 + 동기화 버그 이력 | **Single Source of Truth** | 상태 지도에서 중복 정체성 발견 |
| 동일 이벤트가 ≥ 2곳에서 제각각 처리 | **Command Pattern** | 이벤트 지도에서 같은 단축키/버튼의 다중 핸들러 |
| 자식이 부모의 내부를 ≥ 3회 직접 호출 | **Event Emission** | 의존 방향 지도에서 하향 관통 다수 |
| 순환 의존 | **Event Emission** (우선) | 그래프에서 cycle 탐지 |

---

## 과대설계 필터 (Anti-YAGNI)

다음 조건 중 하나라도 해당하면 패턴을 **제안하지 않는다**. 대신 "현재 구조 유지 권장" 보고서를 작성한다.

- 앱이 한 파일/한 화면 규모 — intent-review로 충분
- 중복 상태가 ≤ 2곳이고 동기화 버그 이력 없음
- 커맨드 총 개수가 ≤ 3개
- 사용자/팀이 현재 구조로 기능 추가에 문제를 겪고 있지 않다
- 관찰 도구(테스트/스냅샷)가 전무한 상태에서 대공사 방향 제안 금지 — 먼저 관찰 도구를 갖추게 권고

---

## 방향 제안 (Direction)

이 스킬의 최종 출력은 **"어느 방향으로 이주해야 하는가"** 까지다. 구체적인 단계 쪼개기, 커밋 계획, Golden Master 시나리오 작성은 **플랜 작성 스킬**(예: `superpowers:writing-plans`)이 뒤이어 받아서 실행 가능한 plan으로 변환한다.

### 방향 제안에 포함되는 것

- **선택된 패턴 이름** — SSOT / Command Pattern / Event Emission 중 하나
- **적용 범위** — 어떤 모듈/파일/호출 지점이 대상인가
- **개괄적 이주 접근** — 한 문단으로 "뼈대 도입 → 호출 이전 → 레거시 제거"의 흐름 설명 (구체 단계 나열 금지)
- **주의점** — 이주 시 가장 위험한 지점, 병행 기간의 주의사항
- **Golden Master 필요 여부** — 필요하다고만 표시, 시나리오 목록 작성은 플랜 스킬이 담당
- **대기 리스트** — 이번 라운드에서 다루지 않을 나머지 증상

### 방향 제안에 포함되지 않는 것 (플랜 스킬 영역)

- ❌ 단계별 실행 가능한 task 목록
- ❌ 커밋 메시지 템플릿 및 커밋 수 예측
- ❌ 파일별 변경 사항 상세
- ❌ Golden Master 시나리오 구체 목록
- ❌ 각 단계별 검증 절차 상세
- ❌ 리스크 레벨 산정

이런 내용을 보고서에 쓰지 않는다. 쓰면 플랜 스킬과 역할이 중복되고 실제 실행 시 두 출처의 불일치가 생긴다.

### 플랜 스킬로의 인계 문구

진단 보고서 마지막 섹션은 항상 다음으로 끝난다:

```markdown
## 다음 단계 — 플랜 작성

이 방향 제안을 실행 가능한 plan으로 변환하려면 `superpowers:writing-plans` 스킬을 호출하세요.
플랜 스킬이 다음을 작성합니다:
- 라운드별 독립 plan 파일
- 단계별 task + 검증 절차
- Golden Master 시나리오 구체 목록
- 커밋 계획

본 방향 제안 문서를 플랜 스킬의 입력으로 전달하면 됩니다.
```

## 보고서 — 진단 (최종 출력)

**저장 위치**: `docs/reviews/architecture-review-YYYY-MM-DD.md`

이 스킬이 생성하는 유일한 문서다. 실행 결과 문서는 플랜 스킬 또는 실행 스킬의 영역이다.

```markdown
# Architecture Review — YYYY-MM-DD

## 컨텍스트
(Phase 0.5로 생성된 자동 추론 컨텍스트 블록 삽입, 또는 service-principles.md 참조 표시)

## 요약
- 진단 대상: {파일 수, 라인 수}
- 주요 증상: {Top 1}
- 권장 패턴: {1개} (또는 "현재 구조 유지")
- 플랜 스킬 호출 필요 여부: {Yes / No (현상 유지)}

## 상태 지도
| 값의 정체성 | 쓰기 지점 | 읽기 지점 | 중복 여부 |

## 이벤트 지도
| 이벤트 | 핸들러 위치 | 해석 일관성 |

## 의존 방향 지도
(문제 있는 관계만 발췌)
- A.render() → B.store.items.push()  ← 내부 관통
- C ↔ D  ← 순환

## 선택된 패턴 및 근거
왜 이 패턴인가 / 왜 지금인가 / 적용 시 영향 범위 개괄

## 방향 제안
- **패턴**: {SSOT / Command / Event Emission}
- **적용 범위**: {대상 모듈/파일/호출 지점 목록}
- **개괄 접근**: 한 문단으로 흐름 서술 (뼈대 → 이전 → 제거의 방향성). 단계 쪼개기 금지.
- **주의점**: 가장 위험한 지점, 병행 기간의 주의사항
- **Golden Master 필요**: Yes/No (필요하다고만 표시)

## 대기 리스트 (이번 라운드 적용 안 함)
- {발견했지만 이번에 다루지 않는 증상들}
- 다음 라운드에서 우선순위 재평가

## 다음 단계 — 플랜 작성

이 방향 제안을 실행 가능한 plan으로 변환하려면 `superpowers:writing-plans` 스킬을 호출하세요.
본 문서를 입력으로 전달하면 플랜 스킬이 라운드별 plan, 단계별 task, Golden Master 시나리오,
커밋 계획까지 작성합니다.
```

## 자가 검증 체크리스트 (보고서 제출 전)

- [ ] 제안한 패턴이 **하나뿐**인가? (복수 패턴 혼합 금지)
- [ ] 과대설계 필터를 통과했는가? (YAGNI 재검토)
- [ ] 보고서에 **단계별 task, 커밋 수, Golden Master 시나리오 목록이 들어가지 않았는가**? (이것들은 플랜 스킬 영역)
- [ ] 방향 제안이 "개괄 한 문단" 수준에 머물렀는가? (구체 단계 쪼개기 금지)
- [ ] 코드를 수정하지 않는다는 원칙을 지켰는가?
- [ ] 새 구조의 함수/이벤트 매개변수가 ≤ 3개인가?
- [ ] 대기 리스트가 명시되어, 나머지 증상이 버려지지 않았는가?
- [ ] 테스트/관찰 도구가 전무한 앱에 대공사 방향을 제안하고 있지 않은가?
- [ ] 마지막 섹션에 **플랜 스킬 호출 안내**가 포함됐는가?

---

## intent-review와의 관계

- **intent-review**: 함수 내부의 이름/구현 괴리 → **명명 추출**로 해결
- **architecture-review**: 함수들 사이 관계 문제 → **경로 재설계**로 해결
- intent-review 보고서에 "분산된 의도 ≥ 3건" 출현 시 이 스킬 권장 트리거 발동
- 이 스킬이 이주를 마친 후 새로 생긴 함수들에 대해 intent-review가 다시 이름을 검증

두 스킬은 순환적으로 보완한다. 한 스킬만으로는 해결되지 않는 문제가 두 스킬의 연계로 해결된다.

## 원칙 요약

| 원칙 | 설명 |
|------|------|
| **경로에 이름** | 데이터/이벤트가 흐르는 길에 이름을 붙여야 예측 가능해진다 |
| **한 번에 하나** | 한 라운드에 하나의 패턴만 제안 |
| **진단까지만** | 스킬은 진단 + 방향 제안까지. 단계 쪼개기와 실행 plan은 플랜 스킬 영역 |
| **과대설계 회피** | 증상이 없으면 패턴도 없다 |
| **코드 미수정** | 이 스킬은 어떤 경우에도 코드를 수정하지 않는다 |
