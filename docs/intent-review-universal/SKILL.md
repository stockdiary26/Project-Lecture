# Intent Review — 함수 의도 vs 구현 검토

> 스크립트/Hook 자동화가 필요하면 같은 폴더의 `SETUP.md`를 참조하여 설정한다.

## 핵심 철학

> **"예측 불가능한 코드는 오해의 근본 원인이다."**

함수명은 약속이다. 함수명과 실제 동작의 괴리(Gap)가 클수록, AI와 개발자는 코드를 오해하고 버그를 양산한다.

**핵심 질문**: "함수명만 보여주고 구현을 상상했을 때, 상상과 실제가 일치하는가?"

---

## 예측 가능성(Predictability) 평가 기준

### CQS (Command-Query Separation)

- **Query (질문)**: 값을 반환하며, 관찰 가능한 상태를 변경하지 않는다.
  - *허용*: memoization, read-through cache, lazy loading, 비침투적 logging
- **Command (명령)**: 시스템 상태를 영구적으로 변경하며, 값을 반환하지 않는다.
- **규칙**: Query가 호출자가 인지할 수 있는 시스템 상태(DB, 전역변수 등)를 몰래 변경하면 '사기꾼(⭐)'이다.

### 별점 체계

| 별점 | 등급 | 설명 | 자동 탐지 힌트 |
|:---:|:---:|---|---|
| ⭐ | 사기꾼 | Query인 척하며 Command 수행 (CQS 위반) | `return`이 있는 함수 내부에서 상태 변경(DB write, 전역 변수 수정) 발견 |
| ⭐⭐ | 과장광고 | 함수명에 없는 중요 부수 효과 존재 | 반환 타입이 void인데 함수명에 없는 상태 변경 로직 존재 |
| ⭐⭐⭐ | 애매모호 | 너무 포괄적인 네이밍 | 범용 동사 단독 사용, 접속사 함수명 |
| ⭐⭐⭐⭐ | 정직함 | 대부분의 로직이 이름에서 유추 가능 | `동사+목적어` 형태, 내부 로직이 그 범위 내 |
| ⭐⭐⭐⭐⭐ | 투명함 | 이름이 곧 구현이며 부작용 없음 | 순수 함수에 가깝거나, 명백한 단일 책임만 수행 |

### 추출 가치 등급

각 후보를 평가할 때 별점과 함께 추출 가치를 판정한다.

| 등급 | 기준 |
|------|------|
| **high** | DRY 위반(중복), 도메인 규칙이 인라인에 숨어 있음, 또는 같은 의도가 여러 곳에 분산 구현됨 |
| **medium** | 복잡도 높지만 1회성 사용 |
| **low** | 현재 위치가 더 읽기 쉬움 |
| **skip** | False positive — 추출하지 말 것 |

## 경고 신호 (Red Flags)

### 1. 인지 부하 과다
- **함수 길이**: 50줄 이상 주의(황), 80줄 이상 위험(적), 100줄 이상 즉시 분해(God Function)
- **논리적 깊이**: 들여쓰기 3단계 이상
- 이유: 작업 기억(Working Memory) 용량 초과

### 2. 접속사가 필요한 함수명
- `And`, `With`, `Also`가 필수적이라면 단일 책임 원칙(SRP) 위반
- 예: `validateAndSaveUser()` → `validate()` + `save()` 분리

### 3. 범용 동사 단독 사용 (Core Domain 한정)
- **금지어**: `process`, `handle`, `manage`, `check`, `do`, `run`, `execute`
- 구체적 동작으로 대체: `calculate`, `validate`, `persist`, `dispatch`
- **예외**: Boundary/Infrastructure 레이어에서는 허용 (`handleHttpRequest`, `processQueueMessage`)

### 4. 함수 내부 다중 관심사
- 함수 내부에 섹션 구분 주석이 2개 이상이면 복수 책임 의심
- 단, Factory Pattern은 예외 (아래 False Positive 참조)

### 5. 분산된 의도 (Scattered Intent)
- 같은 도메인 규칙이 여러 파일/함수에서 각각 인라인으로 구현되어 있음
- 탐지 방법: 유사한 조건문, 같은 필드 조합에 대한 검증, 비슷한 계산 로직이 2곳 이상에서 발견
- 위험: 한쪽만 수정하면 나머지는 옛 로직으로 동작 → 일관성 붕괴
- 조치: 명명 함수로 통합 후 단일 소스로 호출

## 네이밍 가이드

### 추상화 수준 맞추기 (SLAP)
함수 내의 모든 코드는 동일한 추상화 레벨이어야 한다. 비즈니스 로직과 저수준 루프가 섞이지 않도록.

### 도메인 언어 우선 (Ubiquitous Language)
- 기술적 정확성보다 **도메인 의미 전달**이 우선
- Bad: `updateUserStatusToInactive()` → Good: `banUser()`
- 기획서나 현업에서 쓰는 용어를 코드에 그대로 사용

### 의도 명명 함수 (Intent-Naming Functions)
- 인라인 로직이 **도메인 규칙**을 표현할 때는 명명 함수로 추출
- 판별 기준: "이 코드를 주석으로 설명해야 한다면 → 함수로 추출"
- 함수명이 규칙을 약속으로 못박아 리팩토링 시 유실 방지

### Dispatch Table (타입 분기 통합)
`if type === 'A' / 'B' / 'C'` 체인이 여러 곳에 반복되면 테이블로 통합.

```
// Before — 여러 곳에 반복
if (type === 'A') { ... }
else if (type === 'B') { ... }

// After — 명시적 매핑
const HANDLERS = { A: handleA, B: handleB }
```

## False Positive 필터 (추출하지 말 것)

다음 패턴은 경고로 탐지될 수 있지만 **의도적으로 유지해야 한다**:

1. **응집된 순차 로직** — 여러 스타일 설정이 한 함수에 있어도 하나의 상태 전이를 구성하면 OK (도메인 계산만 분리)
2. **1회성 상수 계산** — `const area = w * h` 같은 단순 계산을 굳이 추출하지 않음
3. **프레임워크 관례** — `init()`, lifecycle 메서드, 엔트리포인트 등 관례상 이름이 정해진 것
4. **테스트 파일** — 가독성이 우선. 테스트 코드에는 적용하지 않음
5. **선언 모음** — 상수/아이콘/설정 선언이 나열된 파일의 "긴 함수" false positive
6. **렌더링 순차 호출** — DOM/UI 빌더에서 자식 요소 여러 개를 순서대로 추가하는 패턴
7. **Factory Pattern** — `createXxx()` 함수가 여러 섹션을 가져도 공유 클로저 상태로 결합되어 있으면 유지
   - 판정: 섹션들이 **공유 변수**를 읽고 쓰는가? → YES면 유지, NO면 분리

## 판별 플로우차트: 인라인 vs 함수화

```
코드 블록 발견
    │
    ├── 주석으로 "왜"를 설명해야 하나?
    │       YES → 함수 추출 (함수명 = 주석의 요약)
    │       NO  → 인라인 유지 고려
    │
    ├── 같은 로직이 2곳 이상 있나?
    │       YES → 공유 유틸로 추출
    │       NO  → 다음 질문
    │
    ├── 도메인 규칙/제약을 표현하나?
    │       YES → 함수 추출 (리팩토링 시 유실 방지)
    │       NO  → 다음 질문
    │
    ├── 순수 함수로 만들 수 있나?
    │       YES → 추출해서 테스트 가능하게
    │       NO  → 클로저 내부 유지 (factory pattern 가능성)
    │
    └── 함수 추출 시 매개변수가 4개 이상?
            YES → 구조 재설계 (객체 파라미터 또는 다른 단위로 묶기)
            NO  → 추출 OK
```

## 안전 원칙

> **"테스트 없는 리팩토링 금지."**

### Golden Master 전략
단위 테스트 작성이 불가능한 레거시 코드라면:
1. 현재 코드의 입출력 스냅샷을 찍어 파일로 저장
2. 수정 후 스냅샷과 100% 일치하는지 비교
3. 동작 불변을 보장한 뒤 커밋

### 기계적 분리 금지 (Anti-Pattern: Ravioli Code)
- 줄 수를 줄이기 위해 맥락 없이 코드를 잘라내지 마라
- **응집도(Cohesion)가 기준이다**
- Bad: `part1()`, `part2()` → Good: `calculateTax()`, `applyDiscount()`

## 워크플로우

### 사용 시나리오

| 상황 | 방법 | 비용 |
|------|------|------|
| 함수 수정 직후 자가 검증 | PostToolUse hook 자동 실행 또는 Claude 직접 판단 | 낮음 |
| 커밋 전 자동 경고 | pre-commit hook (비차단) | 낮음 |
| PR/릴리스 전 변경 추세 | `delta-check.sh` 또는 Claude 비교 분석 | 중간 |
| 전수 스캔 | `scan.sh` → Claude AI 평가 | 높음 |

### 3단계 프로세스

**Phase 1: 탐지** (스크립트 또는 Claude 직접)
- 같은 폴더에 `scan.sh`가 있는지 확인한다.
- 스크립트가 있으면: 실행 → 후보 목록 생성 (토큰 절약)
- 스크립트가 없으면: Claude가 대상 디렉토리를 직접 읽고 경고 신호 기반으로 후보 식별
- 스크립트가 없으면 사용자에게 생성을 제안한다 (`SETUP.md` 참조).

**Phase 2: AI 평가**

각 후보를 두 가지 관점에서 평가한다:

**A. 개별 함수 평가** — 별점 + 추출 가치:

```markdown
| 파일:라인 | 함수명 | 별점 | 실제 책임 (1줄) | 추출 가치 |
|-----------|--------|------|-----------------|-----------|
```

**B. 분산 의도 탐지** — 후보들 사이에서 같은 의도의 분산 구현을 찾는다:
- 유사한 조건문/검증 로직이 여러 파일에 있는가?
- 같은 도메인 개념(재고 확인, 권한 검증, 가격 계산 등)이 각각 다르게 구현되어 있는가?
- 발견 시 추출 가치를 **high**로 올리고, 통합 대상으로 묶어서 보고한다.

**Phase 3: 보고서 작성**

```markdown
# Intent Review — YYYY-MM-DD

## 요약
- 스캔: {N}개 파일, {M}개 함수
- 후보: {K}건 (high {X} / medium {Y} / low {Z})
- 최대 이슈: {1줄 요약}

## 🔴 High Priority
### 1. `functionName()` — ⭐⭐ — file:line
**실제 책임**: {한 줄}
**문제**: {한 줄}
**추출 제안**: (before → after 코드)

## 🟡 Medium / 🟢 Low
(표 형식)

## 패턴 분석
- **분산된 의도**: {같은 도메인 규칙이 여러 곳에 각각 구현된 사례}
  - 발견 위치: {파일:라인 목록}
  - 통합 제안: {명명 함수명 + 배치 위치}
- **표면적 중복**: {코드가 복붙된 사례}
- **새 모듈 후보**: {관련 함수들 묶음}

## 아키텍처 제안
{1-3개}

## 다음 단계 권장
(조건부: "분산된 의도" ≥ 3건 또는 "별점 ⭐" 함수가 같은 모듈에 ≥ 2건인 경우에만 출력)

이번 검토에서 분산된 의도가 {N}건 발견되었습니다. 이는 함수 단위를 넘어선 **아키텍처 차원의 경로 설계 문제**일 가능성이 있습니다.

→ 다음 단계로 `/architecture-review` 커맨드 실행을 권장합니다.
조건 미충족 시 이 섹션은 생략합니다.
```

**저장 위치**: `docs/reviews/intent-review-YYYY-MM-DD.md`

### 반복 주기
- **즉시**: 대규모 리팩토링 전
- **주기적**: 릴리스 주기마다
- **트리거**: "이 코드가 왜 이렇게 됐지?" 혼란 발생 시

## 자가 검증 (보고서 완성 전 필수)

- [ ] 각 제안이 정말 **버그 예방** 또는 **도메인 명확성**에 기여하는가?
- [ ] 추출 후 **매개변수가 3개 이하**인가? (아니면 구조 설계 재검토)
- [ ] 제안 함수명이 **1-2 단어**로 의도를 전달하는가?
- [ ] 리팩토링 **영향 범위**가 파악됐는가?
- [ ] False positive 필터를 통과했는가?
- [ ] 같은 의도의 **분산 구현**을 모두 식별했는가? (하나만 추출하고 나머지를 놓치면 불일치 지속)
