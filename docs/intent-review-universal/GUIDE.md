# Intent Review — 왜 이 스킬이 필요한가

## 문제: 디버그 지옥

AI와 함께 코드를 작성하다 보면 이런 순환에 빠지기 쉽다:

1. **버그 발생** → 원인을 추적하고 수정한다
2. **수정 코드를 넣을 곳이 없다** — 이 수정의 "의도"를 담을 공간이 아키텍처에 정의되어 있지 않다
3. **기존 함수 안에 끼워넣는다** — 함수명과 무관한 로직이 함수 내부에 숨어든다
4. **AI가 정리해버린다** — "이 코드는 함수 목적과 맞지 않는다"고 판단하고 제거하거나 단순화한다
5. **같은 버그가 다시 발생한다**

이것이 디버그 지옥이다. 고친 코드가 계속 사라진다.

## 원인: 이름 없는 의도는 생존하지 못한다

AI는 코드를 읽을 때 함수명과 시그니처를 기준으로 판단한다. 함수의 이름이 약속하는 범위 밖의 코드는 "불필요한 코드"로 분류된다.

```
// processOrder() 안에 숨어든 디버깅 수정
function processOrder(order) {
  validate(order)
  // [중요] 2024-03 재고 동기화 타이밍 이슈로 추가
  // 주문 처리 전에 반드시 재고를 다시 확인해야 함
  if (inventory.get(order.itemId) < order.quantity) {
    throw new InsufficientStockError(order.itemId)
  }
  save(order)
  notify(order)
}
```

AI가 이 함수를 리팩토링하면 주석 달린 코드를 `validate()`에 통합하거나 제거할 수 있다. 주석은 약속이 아니기 때문이다.

## 해법: 의도에 이름을 붙여라

```
function processOrder(order) {
  validate(order)
  ensureStockAvailable(order)  // ← 이름이 곧 약속
  save(order)
  notify(order)
}
```

`ensureStockAvailable`이라는 이름이 붙는 순간:
- AI는 이 함수를 "목적이 명확한 코드"로 인식한다 — 제거 대상이 아니다
- 리팩토링 시 이 함수의 존재를 보존해야 한다는 신호가 된다
- 함수가 독립되면, 이 의도가 아키텍처 상 어디에 있어야 하는지를 자연스럽게 묻게 된다

이것이 **의도 명명 함수(Intent-Naming Function)** 패턴이다.

> "이 코드를 주석으로 설명해야 한다면, 함수로 추출하라."
>
> 주석은 사라질 수 있지만, 함수명은 약속이다.

## 또 다른 문제: 의도의 분산

이름 없는 의도가 하나의 함수에 숨어드는 것만이 문제가 아니다. **같은 의도가 여러 곳에서 제각각 구현되는 것**도 위험하다.

```
// 주문 처리에서의 재고 확인
if (inventory.get(order.itemId) < order.quantity) { ... }

// 장바구니에서의 재고 확인
if (stock[item.id] <= 0) { ... }

// 관리자 페이지에서의 재고 확인
if (!checkAvailability(productId)) { ... }
```

같은 "재고 확인"이라는 의도가 세 곳에서 미묘하게 다른 조건으로 구현되어 있다. 한쪽의 조건을 수정하면 나머지 둘은 여전히 옛 로직으로 동작한다.

이것은 단순한 코드 중복이 아니다. **의도가 이름을 갖지 못했기 때문에 각 장소에서 독립적으로 자생한 결과**다.

`ensureStockAvailable()`이라는 명명 함수가 하나 존재했다면:
- 세 곳 모두 이 함수를 호출했을 것이다
- 조건 변경 시 한 곳만 수정하면 된다
- grep 한 번에 이 의도의 모든 사용처를 찾을 수 있다

> 중복 코드는 종종 "이름 없는 의도가 여러 곳에서 자생한 흔적"이다.

## 이 스킬이 하는 일

**Intent Review**는 코드에서 두 가지 위험을 찾아낸다:

1. **숨은 의도** — 함수명이 약속하는 것보다 과도한 책임을 가진 코드
2. **분산된 의도** — 같은 의도가 여러 곳에서 각각 다르게 구현된 코드

그리고:

3. **평가**: 그 괴리가 얼마나 위험한지 별점으로 등급을 매긴다
4. **제안**: 의도를 명명 함수로 통합하고, 아키텍처 상 적절한 위치를 제안한다

리팩토링 도구가 아니다. **의도가 안전하게 살아남을 수 있는 구조를 만드는 시작점**을 찾는 도구다.

## 누구를 위한 스킬인가

- AI 코딩 도구(Claude Code, Copilot 등)와 함께 코드를 작성하는 개발자
- 디버깅 수정이 반복적으로 사라지는 경험을 한 적 있는 개발자
- 코드베이스가 커지면서 "이 코드가 왜 여기 있는지" 설명할 수 없는 함수가 늘어나고 있는 팀

## 어떻게 시작하는가

### 가장 간단한 방법 (설정 없이)

Claude Code에서:
```
intent-review 돌려줘
```

Claude가 코드를 직접 읽고, 함수명과 실제 구현 사이의 괴리를 평가하고, 보고서를 작성한다.

### 자동화하려면

`SETUP.md`를 참조하여:
1. 프로젝트에 맞는 스캔 스크립트를 생성한다 (토큰 절약)
2. PostToolUse Hook을 설정한다 (코드 수정 시 자동 검사)

스크립트가 없어도 스킬은 동작한다. 스크립트는 비용 최적화다.

## 핵심 원칙 요약

| 원칙 | 설명 |
|------|------|
| **함수명은 약속이다** | 이름이 약속한 범위 밖의 코드는 AI에게 제거 대상이 된다 |
| **주석이 아니라 함수로** | 주석은 사라질 수 있지만, 함수명은 코드에 박힌 약속이다 |
| **의도를 발견하면, 공간을 만들어라** | 의도 명명 함수를 만든 후, 아키텍처 상 어디에 배치할지 판단한다 |
| **같은 의도는 한 곳에** | 분산된 의도는 일관성을 파괴한다. 명명 함수로 통합하라 |
| **테스트 없는 리팩토링 금지** | 의도를 분리할 때 동작이 변하지 않음을 보장해야 한다 |
