# Service Design — 왜 이 스킬이 필요한가

## 코드에 없는 정보 문제

`architecture-review`는 **코드만 읽으면** 되는 작업이다. 상태 중복, 이벤트 분산, 의존 그물망은 AST를 훑으면 드러난다. 그래서 패턴 매칭으로 자동화가 된다.

하지만 진짜 아키텍처 설계에는 **코드에 없는 정보**가 반드시 필요하다:

- 서비스가 무엇을 하는가 — 도메인 모델
- 지금 어떤 기능이 있는가 — 현재 기능 인벤토리
- 앞으로 무엇이 들어올 것인가 — 로드맵
- 어디가 확장 축인가 — 사용자/컨텐츠/협업/실시간/디바이스
- 어떤 제약이 있는가 — 팀, 스택, 배포, 예산

이 정보들은 **기획자/개발자의 머릿속과 기획 문서에 있지, 코드 안에 있지 않다.** 코드만 읽고 설계를 제안하는 AI는 자신감 있게 만들어내는 틀린 설계를 생산한다.

그래서 이 스킬은 **인터뷰로 시작**한다. 인터뷰 단계를 생략할 수 없다.

## 인터뷰 프로토콜

### 5단계, 각 단계마다 승인 후 진행

**단계 1 — 서비스 정체성**

- 한 줄 정의: "이 서비스는 _____를 위해 _____를 하는 도구다"
- 핵심 사용자 페르소나 1-2개
- 사용자가 이 서비스에 오는 주요 동기 (1-3개)

이 한 줄이 뒤의 모든 결정의 기준이 된다. 흐릿하면 뒤 단계가 전부 흐릿해진다.

**단계 2 — 도메인 모델**

- 서비스가 다루는 "실제 세계의 것들" (엔티티): 유저, 프로젝트, 문서, 주문 등
- 각 엔티티 간의 관계 (1:N, N:M)
- 엔티티의 생애 주기 (만들어짐 → 변경됨 → 사라짐)

🟡 이 단계에서 "도메인 모델" 개념 이해가 필요하다. `KNOWLEDGE-GAPS.md#domain-model` 참조.

**단계 3 — 현재 기능 인벤토리**

- 현재 제공 중인 주요 기능 5~10개
- 각 기능이 조작하는 엔티티와 이벤트
- 기능 간 공유 자원 (공통으로 쓰는 상태, 이벤트)

인벤토리가 5개 미만이면 설계 논의 자체가 아직 시기상조다. 스킬이 이를 판별하여 "현 시점 설계 불필요" 의견을 낼 수 있다.

**단계 4 — 로드맵 (3~6개월)**

- 추가 예정 기능 3~7개
- 각 기능이 건드릴 엔티티/이벤트
- 불확실한 기능 (넣을지 말지 모름)은 별도 표기

로드맵이 **패턴 선택을 바꾼다.** 예:

- 지금 단순 SSOT가 맞지만 "실시간 협업" 예정 → CRDT 기반 저장소로 상향
- 지금 Command Pattern이 맞지만 "매크로 녹화" 예정 → 직렬화 가능 Command로 상향

**단계 5 — 확장 축과 제약**

- 확장 축 선택: 사용자 수 / 컨텐츠 종류 / 동시 협업 / 실시간성 / 디바이스 종류 (1-2개)
- 기술 스택 자유도 (락인 있는가?)
- 팀 규모 및 숙련도
- 배포 제약 (서버리스? 정적 호스팅? 자체 호스팅?)
- 예산/시간 제약

## 지식 경고 시스템 — 이 스킬의 핵심

**AI는 결정을 대신 내리지 않는다.** 대신 결정 지점마다 "이 결정을 잘하려면 알아야 할 것"을 명시한다.

### 세 단계

| 색 | 의미 | AI 행동 |
|---|---|---|
| 🔵 | 알아두면 좋은 개념 | 보고서에 각주로 기록, 진행 |
| 🟡 | 이 결정에 개념 이해가 필요 | 짧게 설명 후 사용자 확인 요청, 진행 |
| 🔴 | 개념 없이 결정하면 설계 깨짐 | **정지.** 두 선택지 제시 후 대기 |

### 🔴가 뜰 때

AI는 반드시 사용자에게 두 길을 제시한다:

1. **"개념을 먼저 이해하고 오겠다"** → `KNOWLEDGE-GAPS.md` 항목 및 외부 자료 안내, 인터뷰 일시 중단
2. **"지금은 안전한 기본값으로 가겠다"** → 기본값 적용, 결정 문서에 "후속 검토 항목 ⚠️" 태그

두 번째 길을 골라도 괜찮다. 단, 설계 원칙 문서에 **"이 결정은 학습 전 임시 결정임"**이 명시된다. 나중에 이해가 생겼을 때 되돌아볼 수 있게.

### 흔히 🔴가 뜨는 지점

- 여러 화면이 같은 값을 실시간으로 봐야 하는가? → **일관성 모델**
- 사용자 행위를 "상태 변경"으로 저장할지 "발생한 이벤트"로 저장할지 → **상태 vs 이벤트 사고**
- 누가 무엇을 볼 수 있고 할 수 있는가의 복잡도 → **권한 모델**
- 클라이언트에서 계산할지 서버에서 계산할지 → **신뢰 경계**

## 출력: 설계 원칙 문서

인터뷰가 끝나면 `docs/design/service-principles.md` 하나가 생성된다. 이게 스킬의 진짜 산출물이다.

문서 구성:

1. **서비스 정체성** — 한 줄 정의, 핵심 페르소나
2. **도메인 맵** — 엔티티 목록 및 관계도 (텍스트 기반)
3. **현재 기능 인벤토리** — 기능별로 다루는 엔티티/이벤트
4. **로드맵 흡수** — 예정 기능이 도메인 맵에 미치는 영향
5. **확장 축** — 앞으로 복잡도가 몰릴 방향
6. **결정 목록** — 내린 구조적 결정들 (각각 근거 + 🔵🟡🔴 태그)
7. **후속 검토 항목** — 임시로 결정한 것, 학습 후 재검토할 것
8. **architecture-review로의 연결** — 이 원칙에 맞는 현재 구조/패턴 추천

## 두 가지 모드

### 학습자 모드 (Beginner)

- 인터뷰 질문 수 축소 (5단계 × 2~3 질문)
- 결정 지점에서 🔴 발생 빈도가 높을 것을 전제, 기본값 위주로 안내
- 로드맵 대신 "추가하고 싶은 기능 1개"만 받음
- 출력 문서 간소화 (1페이지 이내)

### 개발자 모드 (Full)

- 5단계 × 전체 질문
- 도메인 모델링 세션 수준의 깊이
- 이벤트 스토밍 힌트 제공 (선택)
- 출력 문서 전체 생성

사용자가 시작 시 명시하지 않으면 AI가 응답 품질을 보고 자동 판단한다 (단계 1의 답이 흐릿하면 자동으로 학습자 모드).

## 무엇을 결정하고 무엇을 하지 않는가

### 이 스킬이 결정에 관여하는 것

- 도메인 경계와 모듈 분할의 방향
- 상태 관리 모델 (중앙 집중 / 분산 / 이벤트 기반)
- 이벤트/명령 디스패치 전략
- 확장 축에 맞는 우선 패턴 선택

### 이 스킬이 **결정하지 않는 것**

- 구체적 라이브러리 선택 (Zustand vs Jotai vs Redux 등)
- 팀 협업 방식 (트렁크 기반? GitFlow?)
- 브랜딩/UX 결정
- 인프라 선택 (AWS vs GCP, 자체 호스팅 vs 서버리스)

이것들은 스킬이 **선택지와 트레이드오프**를 제시하지만 결정은 당신이 한다.

## architecture-review와의 연계

- service-design의 출력 `service-principles.md` → architecture-review의 판단 기준
- architecture-review는 코드를 읽을 때 이 원칙 문서를 먼저 참조
- 원칙에서 벗어난 구조 발견 시 이주 제안
- 원칙 자체가 바뀌어야 할 때는 다시 service-design 라운드를 돈다

순환적 관계. 원칙 → 코드 → 원칙 재검토.

## 이 스킬을 쓸 때의 약속

1. **인터뷰 단계를 건너뛰지 않는다.** 코드만 읽고 설계 제안하지 않는다.
2. **결정을 AI가 숨기지 않는다.** 모든 결정은 근거와 지식 요구 수준이 같이 기록된다.
3. **지식 부족 시 "임시 결정"으로 표시한다.** 나중에 되돌아올 수 있게.
4. **학습자에게는 선택지를 좁혀 제시한다.** 결정 피로를 피한다.
5. **설계 원칙 문서는 살아 있는 문서다.** 로드맵이 바뀌면 문서도 바뀐다.
