# Knowledge Gaps — 설계 결정에 자주 필요한 개념 해설

이 문서는 `service-design` 스킬이 결정 지점에서 🟡🔴 경고와 함께 가리키는 개념들을 짧게 설명한다. 각 항목은 **3-5분 읽기** 분량을 목표로 한다. 더 깊이 알고 싶으면 끝의 자습 포인터를 따라가라.

목적: 당신이 이 개념을 이해한 후 설계 결정으로 돌아와서 "후속 검토 항목"의 임시 결정을 확정할 수 있게 하는 것.

---

## domain-model — 도메인 모델

### 한 줄 정의

서비스가 다루는 "실제 세계의 것들"과 그들 사이의 관계를 코드 독립적으로 정리한 그림.

### 왜 이게 설계에 중요한가

도메인 모델이 흐릿하면 코드 속에 같은 개념이 여러 이름으로 돌아다닌다. 한쪽에서는 "유저", 다른 쪽에서는 "멤버", 또 다른 쪽에서는 "계정". 함수를 리팩토링할 때 AI는 이것들이 같은 것인지 다른 것인지 모른다. **분산된 의도**의 근본 원인이 여기다.

### 최소 이해 수준

- **엔티티**: 서비스 안에서 "자기 정체성을 가지는 것" (ID가 있고, 시간이 지나도 같은 것으로 추적됨). 예: 유저, 프로젝트, 문서.
- **값 객체**: 값 자체로 정의되는 것 (같은 값이면 같은 것). 예: 좌표, 색상, 금액.
- **관계**: 1:1, 1:N, N:M. 각 엔티티가 서로 몇 개씩 가질 수 있는지.
- **생애 주기**: 언제 만들어지고, 어떻게 변하고, 언제 사라지는가.

### 자습 포인터

- 검색 키워드: "도메인 주도 설계 입문", "DDD 엔티티 값 객체"
- 3줄 이해로 충분한 경우: 엔티티에 이름/관계/생애를 써보고, 겹치는 것이 있는지만 확인하면 된다.

## state-vs-event — 상태 vs 이벤트 사고

### 한 줄 정의

시스템의 "지금 상태"를 저장할 것인가(상태 기반), "일어난 일들의 기록"을 저장할 것인가(이벤트 기반).

### 왜 이게 설계에 중요한가

단축키 하나만 추가하는 수준에서는 차이 없다. 하지만 "실행 취소(undo)", "히스토리 보기", "동시 편집", "매크로 녹화" 같은 기능이 로드맵에 있으면 이 결정이 전체 구조를 결정한다.

### 두 접근의 차이

**상태 기반 (흔한 방식)**

```
user.balance = 1000
// 송금 후
user.balance = 800
```

지금 값만 남는다. 어제 얼마였는지, 왜 바뀌었는지는 모름.

**이벤트 기반**

```
events = [
  { type: 'deposit', amount: 1000 },
  { type: 'transfer', amount: 200, to: 'Alice' },
]
// balance는 events를 재생해서 계산
```

"일어난 일"이 영구 기록이다. 현재 값은 파생된 결과.

### 결정 힌트

- 단순 기능만 있음 → 상태 기반이 단순하고 빠름
- undo/redo 예정 → 이벤트 기반이 자연스러움
- 실시간 협업 예정 → 이벤트 기반이 거의 필수 (CRDT 등)
- 감사 로그가 중요 → 이벤트 기반

### 자습 포인터

- 검색 키워드: "이벤트 소싱 개념", "event sourcing intro"
- 5분 읽기로 충분: Martin Fowler의 Event Sourcing 글 요약본

## consistency-model — 일관성 모델

### 한 줄 정의

여러 사용자/여러 화면이 같은 데이터를 볼 때, "모두가 즉시 같은 값을 봐야 하는가, 약간 늦어도 되는가"의 선택.

### 왜 이게 🔴인가

이 결정을 모르고 만들면 "가끔 값이 꼬이는" 버그가 발생한다. 디버그가 지옥이다. 재현도 잘 안 된다.

### 두 주요 모델

**강한 일관성 (strong consistency)**

모든 읽기가 최신 쓰기를 반영한다. 느리고 비싸고 협업/실시간에 맞지 않는다. 금융 거래 같은 곳에 필요.

**최종 일관성 (eventual consistency)**

"언젠가는" 모두 같은 값을 본다. 순간적으로는 다를 수 있다. 실시간 협업, 소셜 피드, 채팅 등에 사용.

### 학습자용 기본값

실시간 협업이 없다면 **강한 일관성을 기본값으로** 간주해도 된다 (단일 서버, 단일 DB). 그러면 일관성 문제를 신경 쓸 일이 거의 없다.

실시간 협업이 있다면 **최종 일관성을 받아들여야** 하며, 이를 위한 구조(CRDT, OT, event sourcing)가 필요하다.

### 자습 포인터

- 검색 키워드: "eventual consistency 쉬운 설명", "CAP 이론 개념"
- 바이브코딩 수준에서는 "실시간 협업 있음/없음"만 구분해도 80% 커버된다.

## trust-boundary — 신뢰 경계 (클라이언트-서버 경계)

### 한 줄 정의

어떤 계산을 클라이언트(사용자 기기)에서 하고 어떤 것을 서버에서 할 것인가. 그리고 그 결정이 보안과 직결된다는 사실.

### 왜 이게 🔴인가

"할인율 계산을 클라이언트에서 하면 사용자가 임의로 조작 가능"하다. "권한 검사를 클라이언트에서만 하면 API로 우회 가능"하다. 이걸 모르면 설계가 보안 구멍이 된다.

### 원칙

**클라이언트를 신뢰하지 마라.** 클라이언트에서 한 모든 검증은 "UX 편의"일 뿐이다. 진짜 검증은 서버에서.

### 결정 힌트

- 보여주기만 하는 것 → 클라이언트 OK (UX)
- 결정/변경을 하는 것 → 서버에서 재검증 필수
- 비밀값/API 키 → 절대 클라이언트에 두지 않음
- 할인, 가격, 권한 → 서버 진실

### 정적 호스팅 수준이라면

서버가 없는 경우(순수 정적 SPA)는 "신뢰 경계"가 아예 사용자 안에만 있다. 이때는 "모든 데이터는 공개"라고 간주하고 설계해야 한다. 비밀값이나 다른 사용자 정보를 다루면 안 된다.

### 자습 포인터

- 검색 키워드: "OWASP 입문", "클라이언트 검증 서버 검증"

## auth-model — 권한 모델

### 한 줄 정의

누가 무엇을 볼 수 있고, 누가 무엇을 할 수 있는가를 체계적으로 정의하는 방법.

### 왜 이게 중요한가

권한이 코드 여기저기에 `if (user.role === 'admin')` 식으로 흩어지면, 새 역할 추가 시 모든 곳을 찾아 고쳐야 한다. 그리고 놓치는 순간 보안 구멍.

### 두 주요 접근

**RBAC (Role-Based Access Control)**

사용자에게 역할(관리자, 편집자, 뷰어)을 주고, 역할별로 할 수 있는 일 목록을 정의.

간단하고 대부분의 앱에 충분.

**ABAC (Attribute-Based Access Control)**

"편집자이고, 이 문서의 작성자이고, 근무 시간이고, 사무실 IP에서 접속한 경우"처럼 속성 조합으로 권한을 결정.

복잡하지만 유연. 대형 엔터프라이즈에서.

### 학습자용 기본값

- 사용자 종류가 2-3개 → RBAC가 맞음
- 혼자 쓰는 앱 → 권한 모델 불필요, 건너뜀

### 결정 힌트

- "다른 사용자가 있는가?" → 있으면 권한 모델 필요
- "역할이 몇 가지?" → 2-3개면 RBAC, 더 복잡하면 RBAC+일부 ABAC

### 자습 포인터

- 검색 키워드: "RBAC 개념", "역할 기반 접근 제어 기본"

## cohesion-coupling — 응집도와 결합도

### 한 줄 정의

**응집도**: 한 모듈 안의 것들이 서로 얼마나 관련 있는가 (높을수록 좋음).
**결합도**: 서로 다른 모듈들이 얼마나 얽혀 있는가 (낮을수록 좋음).

### 왜 이게 중요한가

모듈/폴더를 나눌 때 "어떻게 나눠야 하나"의 기준이 된다. 기준이 없으면 폴더 구조가 "MVC / 타입별"로 피상적으로 나뉘고, 기능 하나 바꾸려면 여러 폴더를 다 건드리게 된다.

### 좋은 분할의 지표

- 한 기능을 바꿀 때 **한 모듈만** 건드리면 된다 → 응집도 높음
- 한 모듈을 바꿔도 **다른 모듈이 안 깨진다** → 결합도 낮음

### 실용적 가이드

도메인/기능 기반 분할이 타입 기반 분할보다 거의 항상 낫다.

나쁜 분할:
```
/components  /services  /utils  /types
```

좋은 분할:
```
/users  /projects  /comments  /shared
```

### 자습 포인터

- 검색 키워드: "응집도 결합도 예제", "feature-based folder structure"

## fail-fast-vs-graceful — 오류 처리 철학

### 한 줄 정의

오류가 났을 때 **즉시 멈출 것인가(fail-fast)**, **부분적으로라도 계속 갈 것인가(graceful)**.

### 왜 이게 중요한가

이 선택이 일관되지 않으면 어떤 화면에서는 오류가 빨갛게 뜨고, 어떤 화면은 조용히 빈 값이 표시되고, 어떤 화면은 앱이 멎는다. 사용자는 앱을 믿을 수 없게 된다.

### 두 접근

**fail-fast**

오류가 나면 즉시 예외를 던진다. 로그 남고, 사용자에게 알린다. 수정 위치가 명확.

적합: 비즈니스 규칙 위반, 예상 외 상태, 개발/테스트 환경.

**graceful**

오류를 흡수하고 기본값이나 부분 결과를 반환한다. 사용자 경험은 매끄럽지만 버그가 숨는다.

적합: 외부 API 실패, 선택적 기능, 프로덕션 환경의 비-핵심 기능.

### 원칙

**"잘못된 상태로 계속 가는 것보다 멈추는 게 낫다."** 기본은 fail-fast. graceful은 의도적으로 선택해야 할 곳에만.

### 자습 포인터

- 검색 키워드: "fail-fast 원칙", "에러 바운더리 react"

## test-pyramid — 테스트 피라미드

### 한 줄 정의

테스트를 세 층으로 구성한다: 단위 (많이) → 통합 (중간) → E2E (적게). 아래로 갈수록 많이, 빠르게.

### 왜 이게 중요한가

테스트가 없는 앱에 대해 `architecture-review`가 대공사를 제안할 수 없다. Golden Master 스냅샷이 유일한 안전망이 되는데, 그것만으로는 회귀를 잡기 어렵다.

### 세 층

- **단위**: 함수 하나. 빠르고 많음 (수백~수천 개).
- **통합**: 여러 함수/모듈이 함께. 느림, 중간 수 (수십 개).
- **E2E**: 사용자 관점 전체 흐름. 매우 느림, 소수 (핵심 플로우 5-10개).

### 학습자용 기본값

- 단위 테스트가 전혀 없다면 → E2E 스타일의 "주요 시나리오 체크리스트"라도 만든다 (Golden Master 대체).
- 리팩토링이 예정돼 있다면 → 그 범위만이라도 단위 테스트를 먼저 작성한다 ("테스트 없는 리팩토링 금지" 원칙).

### 자습 포인터

- 검색 키워드: "test pyramid", "테스트 피라미드 개념"

## 이 문서 사용법

1. `service-design` 스킬이 결정 지점에서 🟡🔴 경고를 띄우면 해당 앵커로 온다 (`#domain-model` 등).
2. 3-5분 읽고 자기 서비스에 대입한다.
3. 인터뷰로 돌아가 답을 갱신한다.
4. 학습이 깊이 필요하다면 "후속 검토 항목"으로 임시 결정해두고 나중에 다시 온다.

**목표는 완벽한 이해가 아니다.** 당신이 내리는 결정이 어떤 함의를 가지는지 알고 선택할 수 있을 정도면 충분하다.
